## 3. Backend Modules

### 3.1 Server Module (`inbox/server/`)
**Responsibility:** REST API server handling all CRUD operations, search, enrichment, and job queue management.

**Technology:**
- **Framework:** FastAPI (async, auto-generated OpenAPI docs, Pydantic-native)
- **ORM:** SQLModel (SQLAlchemy 2.0 + Pydantic â€” shared models for DB and API)
- **Database Driver:** asyncpg (async PostgreSQL driver)
- **Migrations:** Alembic (auto-generate from SQLModel metadata)
- **Vector Search:** pgvector extension + SQLAlchemy custom types
- **Full-Text Search:** PostgreSQL tsvector + GIN index
- **Job Queue:** PostgreSQL `job_queue` table with background worker loop
- **Configuration:** Pydantic Settings with `.env` file support
- **ASGI Server:** Uvicorn (HTTP/1.1 + WebSocket support)

**Key Design Decisions:**
- **FastAPI over Flask/Django:** FastAPI provides native async support, automatic OpenAPI generation, and deep Pydantic integration â€” perfect for a project where the same models serve both database and API serialization.
- **SQLModel over raw SQLAlchemy:** SQLModel combines SQLAlchemy ORM with Pydantic validation. The same `Item` model can be used as a database table, API request body, and API response â€” eliminating duplicate schema definitions.
- **Background jobs via PostgreSQL:** A simple `job_queue` table + worker loop instead of Redis/Celery. This avoids introducing additional infrastructure for a single-user app. The worker polls for `pending` jobs and processes them asynchronously.

### 3.2 CLI Module (`inbox/cli/`)
**Responsibility:** Terminal-native interface for rapid capture and feed browsing.

**Technology:**
- **Framework:** Typer (built on Click, type hints, auto-generated help)
- **HTTP Client:** httpx (sync + async, HTTP/2 support)
- **Output Formatting:** Rich (colors, tables, progress bars, pretty JSON)
- **Configuration:** TOML config file at `~/.config/inbox/config.toml`
- **Error Handling:** Rich error messages with suggestions

**Commands:**
| Command | Description | Example |
|---------|-------------|---------|
| `inbox add <content>` | Capture text or URL | `inbox add "Buy milk"` |
| `inbox add --file <path>` | Upload a file | `inbox add -f ~/notes.pdf` |
| `inbox add --stdin` | Read from stdin | `cat notes.md \| inbox add --stdin` |
| `inbox list` | Browse feed (default last 20) | `inbox list -n 50 --type url` |
| `inbox info <id>` | Show item details | `inbox info abc123` |
| `inbox delete <id>` | Delete with confirmation | `inbox delete abc123 --force` |
| `inbox search <query>` | Full-text search | `inbox search "rust tutorial"` |
| `inbox tags` | List tags | `inbox tags` |
| `inbox categories` | List categories | `inbox categories` |
| `inbox stats` | Show statistics | `inbox stats` |
| `inbox open` | Open web UI in browser | `inbox open` |

**Design Decisions:**
- **Typer over Click:** Typer uses Python type hints for argument/option definitions, reducing boilerplate. It also has built-in shell completion generation.
- **httpx over requests:** httpx supports HTTP/2 and has both sync and async APIs. It's the modern standard for Python HTTP clients.
- **Rich over plain print:** Rich provides beautiful terminal output with zero effort â€” tables for `list`, JSON formatting for `--json`, colors for status, and progress indicators for uploads.

### 3.3 Domain / Shared Models (`inbox/models.py`)
**Responsibility:** Centralized data model definitions shared between server and CLI.

**Technology:**
- **Models:** SQLModel (SQLAlchemy 2.0 base + Pydantic v2 validation)
- **Serialization:** Pydantic `model_dump()` / `model_dump_json()` â€” used automatically by FastAPI
- **Validation:** Pydantic field validators (type detection, URL validation, etc.)
- **Type Detection:** Pure Python functions (regex for URLs, mimetype for files, etc.)

**Key Entities:**
| Entity | Fields | Relationships |
|--------|--------|--------------|
| `Item` | id, type, captured_at, raw_text, blob_path, metadata (JSONB), enrichment (JSONB), search_vector (tsvector), embedding (vector), deleted_at | has_many tags, has_many categories |
| `Tag` | id, name, auto_generated, created_at | belongs_to_many items |
| `Category` | id, name, description, created_at | belongs_to_many items |
| `JobQueue` | id, item_id, job_type, status, attempts, max_attempts, run_at, created_at, completed_at, error_message | belongs_to item |

**Design Decisions:**
- **SQLModel over separate Pydantic + SQLAlchemy models:** In the old Rust architecture, a separate `domain` crate held shared structs. In Python, SQLModel lets us use the same class as both a database table and an API schema. This eliminates an entire layer of "DTO mapping" that would otherwise be necessary.
- **Pydantic validators for type detection:** Instead of factory functions in the server, we use Pydantic `@field_validator` decorators on the `Item` model to auto-detect `type` from `raw_text` content.

## 4. Web UI Module (`web/`)

**Responsibility:** Browser-based interface for browsing, search, and management.

**Technology:**
- **Framework:** React 19 + Vite
- **Styling:** Tailwind CSS v4 + Catppuccin theme tokens
- **State:** Zustand (client-side cache of server state)
- **HTTP Client:** Native `fetch` + custom hooks
- **Components:** Reuse from old prototype with API adapter layer

**Architecture Changes from Prototype:**
1. **Storage adapter:** Replace Dexie/IndexedDB with `fetch()` calls to the Python API
2. **State shape:** Zustand store now mirrors server API responses instead of local database schema
3. **Real-time:** Optional WebSocket or Server-Sent Events for live feed updates
4. **Upload:** Direct multipart upload to server instead of browser blob storage
5. **Offline support:** Service Worker with background sync (future)

## 6. Database Architecture

### 6.1 Schema Overview
PostgreSQL with extensions: `pgvector` (vector embeddings), `pg_trgm` (trigram text search).

**Tables:**
- `items` â€” core entity with JSONB metadata, tsvector search, vector embedding
- `tags` â€” user-defined and auto-generated tags
- `item_tags` â€” many-to-many junction
- `categories` â€” broad category buckets
- `item_categories` â€” many-to-many junction
- `job_queue` â€” background enrichment jobs

### 6.2 Indexes
- `items.captured_at` â€” B-tree for chronological feed queries
- `items.type` â€” B-tree for type filtering
- `items.search_vector` â€” GIN for full-text search
- `items.embedding` â€” IVFFlat for vector similarity search
- `tags.name` â€” B-tree for tag lookups
- `categories.name` â€” B-tree for category lookups
- `job_queue(status, run_at)` â€” composite for worker polling
- `job_queue.item_id` â€” B-tree for per-item job lookups

### 6.3 FTS Trigger
`update_search_vector()` trigger automatically rebuilds `tsvector` on INSERT/UPDATE by combining:
- `raw_text` (weight A)
- `metadata.title` (weight B)
- `metadata.description` (weight C)
- `enrichment.summary` (weight C)

## 8. Communication Between Modules

### 8.1 Frontend â†” Server
- **Protocol:** HTTP/1.1 (REST API)
- **Data Format:** JSON
- **Authentication:** None (single-user, local network only)
- **CORS:** Configured for local development (`*` in dev, restricted in prod)
- **Content Negotiation:** JSON only for v0

### 8.2 CLI â†” Server
- **Protocol:** HTTP/1.1
- **Client:** httpx with configurable base URL (`~/.config/inbox/config.toml`)
- **Data Format:** JSON request/response bodies
- **Error Handling:** Rich error display with human-friendly messages

### 8.3 Server â†” Database
- **Protocol:** PostgreSQL wire protocol over TCP
- **Driver:** asyncpg (async)
- **ORM:** SQLModel/SQLAlchemy 2.0 with async session
- **Migrations:** Alembic (synchronous, run against same DATABASE_URL)

## 10. Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Python GIL limits CPU parallelism | Medium | Use async/await for I/O-bound work (DB, HTTP); use `multiprocessing` or `ProcessPoolExecutor` for CPU-bound embedding generation |
| SQLModel is relatively new | Low-Medium | Backed by SQLAlchemy 2.0 (mature); fallback to raw SQLAlchemy if needed |
| sentence-transformers model size (~100MB) | Low | Download once, cache in blob storage; run in background worker |
| Single process for server + worker | Medium | Worker loop runs as asyncio task within same process; can be split to separate process later |
| uv lockfile not yet generated | Low | Run `uv sync` after entering dev shell |
| FastAPI + asyncpg connection pool tuning | Low | Default pool settings are reasonable for single-user; tune `pool_size` if needed |
