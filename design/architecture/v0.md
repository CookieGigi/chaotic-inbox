# Chaotic Inbox — Architecture Proposal v0

**Project:** Chaotic Inbox — Full-Stack Rewrite  
**Status:** Draft  
**Version:** 0.1.0  
**Date:** 2026-05-23  
**Principles:** CLI-first, modular monolith, shared domain, API-driven

---

## 1. Overview

This document proposes an architecture for evolving Chaotic Inbox from a browser-only prototype into a full-stack application with multiple frontends (CLI, Web UI), a backend server, and persistent database storage.

The guiding principles:
- **CLI first**: The CLI is the primary v0 frontend. It is fast to build, forces a clean API design, and serves power users.
- **Modular monolith**: The server is a single deployable unit, but internally decomposed into independent modules (capture, search, enrichment, etc.). We split services only when operational pain demands it.
- **API-driven**: All frontends consume the same REST API. No special server-rendered pages for the web UI.
- **Shared domain**: Business logic (type detection, factories, formatting) lives in a shared library consumed by server, CLI, and (optionally) web UI build pipeline.

## 3. Component Details

### 3.1 CLI (`cmd/inbox/`)

| Property | Value |
|----------|-------|
| Language | Go |
| Distribution | Single static binary per platform (Linux, macOS, Windows) |
| Dependencies | `cobra` (CLI framework), standard library HTTP client |
| Config | `~/.config/inbox/config.toml` + env vars (`INBOX_SERVER`, `INBOX_API_KEY`) |

**Responsibilities:**
- Capture: `inbox add`, pipe support, file upload
- Feed & Browse: `inbox list` with filters
- Search: `inbox search` (full-text + semantic)
- Management: `inbox show`, `inbox edit`, `inbox delete`, `inbox tags`, `inbox categories`
- Configuration: `inbox config`
- Scripting: `--json` output for all commands

**Why Go:**
- Static binary = zero runtime dependencies (critical for CLI tool)
- Fast compilation, cross-compilation built-in
- Standard HTTP client is excellent (no heavy dependencies)
- Tab completion generation supported by `cobra`

---

### 3.2 Web UI (`web/`)

| Property | Value |
|----------|-------|
| Framework | React 19 (SPA, not SSR) |
| Build Tool | Vite |
| Styling | Tailwind CSS v4 |
| State | TanStack Query (server state) + Zustand (client state) |
| HTTP Client | `fetch` + custom API client layer |

**Responsibilities:**
- Replicate old prototype UX: global paste/drop/typing, chronological feed
- Search interface with live results
- Settings panel (accent color, preferences)
- Block rendering (text, URL, image, file) using Design System tokens
- Mobile-responsive layout

**Why SPA + API, not Next.js:**
- The server is API-first, not page-oriented
- No SSR benefits for an authenticated dashboard-style app
- Simpler deployment (static files served by any CDN or nginx)
- Design System is already built for client-side rendering

### 3.4 Database Layer

| Property | Value |
|----------|-------|
| Engine | PostgreSQL 16+ |
| Vector Extension | `pgvector` |
| Migrations | `golang-migrate` or `sqlx` + hand-written SQL |
| Connection Pool | `pgx` (Go PostgreSQL driver) |

**Schema Design:**

```sql
-- Core items table
-- Implements the Data Contract from spec F-04:
--   RAW DATA (immutable), CAPTURE METADATA, BACKGROUND ENRICHMENT, SEARCH INDEXING
-- are stored in separate logical layers even when colocated in one table.
-- See spec F-04 (Persistent Storage) and F-05 (Semantic Search).
CREATE TABLE items (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    type TEXT NOT NULL CHECK (type IN ('text', 'url', 'image', 'file')),
    captured_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),

    -- === RAW DATA (immutable, sacred) ===
    -- Inline raw content for text/url; NULL for blobs (see blob_storage)
    raw_text TEXT,
    -- Blob reference for images/files (immutable after creation)
    blob_path TEXT,

    -- === CAPTURE METADATA (computed at creation time) ===
    -- Type-specific metadata (JSONB for flexibility)
    metadata JSONB NOT NULL DEFAULT '{}',

    -- === BACKGROUND ENRICHMENT (computed async, optional) ===
    -- Enrichment results: title, description, summary, dimensions, tags, etc.
    enrichment JSONB NOT NULL DEFAULT '{}',

    -- === SEARCH INDEXING (computed async, owned by Search module) ===
    -- Full-text search vector (auto-updated via trigger)
    search_vector TSVECTOR,
    -- Embedding vector for semantic search (384-dim)
    embedding VECTOR(384),

    -- Soft delete support (optional for v0)
    deleted_at TIMESTAMPTZ
);

-- Indexes
CREATE INDEX idx_items_captured_at ON items(captured_at);
CREATE INDEX idx_items_type ON items(type);
CREATE INDEX idx_items_search_vector ON items USING GIN(search_vector);
CREATE INDEX idx_items_embedding ON items USING ivfflat (embedding vector_cosine_ops);

-- Tags (many-to-many with items)
CREATE TABLE tags (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,
    auto_generated BOOLEAN NOT NULL DEFAULT false,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE item_tags (
    item_id UUID REFERENCES items(id) ON DELETE CASCADE,
    tag_id INT REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (item_id, tag_id)
);

-- Categories
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,
    description TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE item_categories (
    item_id UUID REFERENCES items(id) ON DELETE CASCADE,
    category_id INT REFERENCES categories(id) ON DELETE CASCADE,
    PRIMARY KEY (item_id, category_id)
);

-- FTS trigger
CREATE OR REPLACE FUNCTION update_search_vector()
RETURNS TRIGGER AS $$
BEGIN
    NEW.search_vector :=
        setweight(to_tsvector('english', COALESCE(NEW.raw_text, '')), 'A') ||
        setweight(to_tsvector('english', COALESCE(NEW.enrichment->>'title', '')), 'B') ||
        setweight(to_tsvector('english', COALESCE(NEW.enrichment->>'description', '')), 'C') ||
        setweight(to_tsvector('english', COALESCE(NEW.enrichment->>'summary', '')), 'C');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER items_search_vector_update
    BEFORE INSERT OR UPDATE ON items
    FOR EACH ROW EXECUTE FUNCTION update_search_vector();
```

**Why PostgreSQL + pgvector:**
- **Relational**: Structured data with JSONB flexibility for metadata/enrichment
- **Full-text search**: Native `tsvector`/`tsquery` — no external search engine needed for v0
- **Vector search**: `pgvector` extension adds production-grade vector indexing (IVFFlat, HNSW)
- **Operational simplicity**: Single database, well-understood backups, proven at scale
- **Self-hostable**: Runs on a single machine; no cloud service lock-in

### 3.5 Blob Storage

| Property | Value |
|----------|-------|
| Default | Local filesystem (`/var/lib/inbox/blobs/`) |
| Configurable | S3-compatible (MinIO, AWS S3, etc.) |
| Organization | Content-addressable or UUID-based paths |

**Storage Strategy:**
- Text/URLs: stored inline in PostgreSQL (`raw_text` column)
- Images/Files: stored as blobs; database holds reference path
- Blob path format: `blobs/{type}/{first-2-chars-of-uuid}/{uuid}`
- Deduplication: optional content-addressable storage (SHA-256 as key)

### 3.6 Background Job Queue

| Property | Value |
|----------|-------|
| v0 Implementation | In-process worker pool + PostgreSQL-backed queue table |
| Future | Redis/RabbitMQ if throughput demands |
| Retry Policy | Exponential backoff: 5s, 25s, 125s (max 3 retries) |
| Dead Letter | Failed jobs moved to `failed_jobs` table after max retries |

**Queue Table:**
```sql
CREATE TABLE job_queue (
    id BIGSERIAL PRIMARY KEY,
    item_id UUID NOT NULL REFERENCES items(id),
    job_type TEXT NOT NULL, -- 'url_enrich', 'image_extract', 'embedding', 'categorize', 'summarize'
    status TEXT NOT NULL DEFAULT 'pending', -- pending, running, done, failed
    attempts INT NOT NULL DEFAULT 0,
    max_attempts INT NOT NULL DEFAULT 3,
    run_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at TIMESTAMPTZ,
    error_message TEXT
);

CREATE INDEX idx_job_queue_pending ON job_queue(status, run_at) WHERE status = 'pending';
```

**Worker Pool:**
- 4-8 concurrent workers (configurable)
- Each worker polls `job_queue` every 1 second
- Jobs are processed independently; failure of one job does not block others
- Embedding generation uses a local ONNX runtime or lightweight model server

### 3.7 Embedding & NLP Pipeline

| Component | Technology | Notes |
|-----------|-----------|-------|
| Embedding Model | `all-MiniLM-L6-v2` (384-dim) | Fast, high quality, runs locally via ONNX |
| ONNX Runtime | `onnxruntime-go` or Python bridge | Go bindings or small sidecar |
| Categorization | Rule-based + lightweight classifier | Rules for v0; ML model in v1 |
| Summarization | Local LLM (Llama 3.2 3B or similar) | Optional background job |
| URL Fetching | Standard HTTP client + HTML parser | Extract `og:title`, `og:description` |

**Pipeline Flow:**
```
Item Created
  ├─→ Immediate: persist to DB, return 201 to client
  ├─→ Job 1 (URL Enrichment): fetch metadata, update enrichment JSONB
  ├─→ Job 2 (Image Extraction): read dimensions, generate thumbnail
  ├─→ Job 3 (Embedding): generate 384-dim vector, update embedding column
  ├─→ Job 4 (Categorization): assign tags/categories
  └─→ Job 5 (Summarization): generate summary for long content (optional)
```

## 6. Data Flow

### 6.1 Capture Flow (CLI)
```
User: inbox add "NixOS flake tips"
  └─→ CLI parses command, builds JSON payload
        └─→ POST /v1/items (raw_text="NixOS flake tips")
              └─→ Server: validate, create Item record
                    ├─→ INSERT into items (type='text', raw_text=..., metadata=...)
                    ├─→ Queue jobs: embedding, categorize
                    └─→ RETURN 201 with item JSON
                          └─→ CLI prints item ID and summary
```

### 6.2 Capture Flow (Web UI)
```
User: Cmd+V anywhere in browser
  └─→ Web UI: paste event handler
        ├─→ Optimistic UI: append block to feed immediately
        └─→ POST /v1/items (multipart or JSON)
              └─→ Server: persist, queue jobs
                    └─→ RETURN 201
                          └─→ Web UI: confirm block (replace optimistic with server data)
                                └─→ Feed scrolls to new block
```

### 6.3 Search Flow
```
User: inbox search "NixOS setup" OR types in web search bar
  └─→ GET /v1/items/search?q=NixOS+setup
        └─→ Server:
              ├─→ PostgreSQL FTS query (fast recall)
              ├─→ pgvector similarity query (semantic re-ranking)
              ├─→ Merge + rank results
              └─→ RETURN ranked items with match context
                    └─→ Client renders results
```

### 6.4 Background Enrichment Flow
```
Worker polls job_queue
  └─→ Finds pending job: "embedding" for item X
        └─→ Fetches item content from DB
              ├─→ Runs ONNX model → 384-dim vector
              ├─→ UPDATE items SET embedding = [...] WHERE id = X
              └─→ Marks job as done
```

## 7. Module Boundaries & Contracts

### 7.1 Internal Module Rules

| Rule | Rationale |
|------|-----------|
| `domain/` has zero external dependencies | Can be imported by CLI, server, and web build tools |
| `api/` depends on `capture/`, `search/`, etc. | Handlers orchestrate module services |
| Modules communicate via Go interfaces, not direct struct access | Enables test mocking and future extraction |
| Database access only via `storage/postgres/` | No raw SQL outside storage layer |
| Queue jobs only enqueued by module services, not handlers | Keeps HTTP layer thin |

### 7.2 Dependency Direction

```
CLI         Web UI
  │            │
  └─────┬──────┘
        │
   ┌────▼────┐
   │   API   │  (HTTP handlers, middleware)
   │  Layer  │
   └────┬────┘
        │
   ┌────▼────┐
   │ Modules │  (capture, search, enrichment)
   │         │
   └────┬────┘
        │
   ┌────▼────┐
   │ Domain  │  (shared types, factories, detection)
   │  Layer  │
   └────┬────┘
        │
   ┌────▼────┐
   │ Storage │  (postgres, blob, cache)
   │  Layer  │
   └─────────┘
```

## 8. Technology Decisions

| Decision | Choice | Alternatives Rejected | Rationale |
|----------|--------|----------------------|-----------|
| Server language | Go | Rust, Python, Node.js | Fast, simple, excellent concurrency, static binary, great PostgreSQL ecosystem |
| CLI language | Go | Python, Rust | Same as server; shared domain library; zero-dependency binary |
| Web framework | React SPA | Next.js, SvelteKit | No SSR needed; API-first; preserve old design system |
| Database | PostgreSQL + pgvector | SQLite, MySQL, MongoDB, Qdrant | Full relational + vector in one engine; proven; self-hostable |
| Job queue | PostgreSQL table | Redis, RabbitMQ | Minimize infrastructure; v0 throughput is low |
| Embedding | ONNX Runtime | Python + Transformers, OpenAI API | Runs locally, no API key, no network dependency |
| Auth v0 | None (single-user) | JWT, API key | Self-hosted default; add API key in v1 |
| File storage | Filesystem | S3 mandatory | Self-hosting default; S3 configurable |

## 9. Scalability Path

| Scale | Changes |
|-------|---------|
| **1 user, 1 server** (v0) | Single binary + PostgreSQL + filesystem. Everything on one machine. |
| **10 users, 1 server** (v1) | Add API key auth. Nginx reverse proxy. Basic rate limiting. |
| **100 users, 1 server** | Redis for caching + job queue. Connection pooling tuning. CDN for web statics. |
| **1000 users, multiple servers** | Extract job workers to separate processes. S3 for blob storage. Load balancer. |
| **10000+ users** | Split modules to services if needed. Read replicas for PostgreSQL. Dedicated vector DB if pgvector limits hit. |

## 11. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Embedding model too slow on CPU | High | Use quantized ONNX model; worker pool limits concurrency; GPU optional |
| PostgreSQL vector index performance degrades | Medium | Monitor with `pg_stat_statements`; add HNSW index if IVFFlat insufficient |
| Large file uploads overwhelm server | Medium | Stream uploads to temp files; size limits (100MB default); chunked upload API |
| CLI binary size too large | Low | Go binaries are small; ONNX runtime is the bulk; optional slim build without NLP |
| Old IndexedDB data migration | Medium | Export JSON from old app; `inbox import` command reads the format |
| Concurrent feed updates in Web UI | Low | Optimistic UI + server ack; WebSocket optional in v1 for real-time |
