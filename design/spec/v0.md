# Chaotic Inbox — Feature Specification v0

**Project:** Chaotic Inbox — Personal Knowledge Capture
**Status:** Draft
**Version:** 0.1.0
**Date:** 2026-05-23
**Scope:** Full-stack rewrite (CLI + Web UI + Server + Database)

---

## 1. Global Introduction

### 1.1 Intent

Chaotic Inbox captures **anything** — text, URLs, files, images — so the user can find it back later. The core promise is **frictionless capture**:

> **Copy/paste or drag/drop → send and forget.**

No loading spinners. No save buttons. No prompts. No folders to choose. No tags to assign. The system captures, stores, and processes in the background while the user moves on.

### 1.2 Philosophy

- **Capture is sacred.** The raw input is never modified, never truncated, never lost.
- **Organization is deferred.** The user throws everything into an append-only pile. The system (and the user, later) imposes order.
- **Speed is trust.** If capture feels slow, the user won't trust it. Every capture must feel instantaneous.
- **Interfaces are multiple.** The same backend serves a fast CLI, a rich web UI, and future interfaces (mobile, browser extension, API).

### 1.3 Input Types

| Type | Example | Capture Methods |
|------|---------|-----------------|
| `text` | "look into Nix flake templates" | Paste, type (CLI or Web), API |
| `url` | `https://github.com/...` | Paste, type, API |
| `image` | Screenshot, photo, diagram | Paste, drag-and-drop, API upload |
| `file` | PDF, markdown, zip, binary | Drag-and-drop, CLI `inbox add`, API upload |

### 1.4 User Journey

```
1. User encounters something they don't want to lose
2. They paste it, drop it, type it, or run a CLI command
3. The system acknowledges immediately (no delay perceived)
4. Background processing begins: type detection, metadata extraction,
   embedding generation, optional enrichment
5. The item appears in the feed / list / search index
6. Later, the user finds it via search, browse, or automatic categorization
```

## 2. Feature Catalog

---

### F-01 — Instant Capture (Universal)

**Priority:** P0
**Applies to:** CLI, Web UI, API

#### Description
Any input method — paste, drop, type, CLI command, API call — results in immediate persistent storage. The user perceives zero latency.

#### Behaviour
- Server persists to database before acknowledging; client updates UI on acknowledgement
- If persistence fails, user sees clear error; no phantom items
- Raw payload is stored byte-for-byte identical; never modified at capture time

#### Acceptance Criteria
- Capture latency perceived as instantaneous (<50ms optimistic UI, <200ms CLI, <100ms API)
- Failed writes surface error to user; no "silent loss"
- Raw payload is byte-for-byte identical in storage

---

### F-02 — Type Detection & Classification (Automatic)

**Priority:** P0
**Applies to:** Server (shared logic)

#### Description
The system automatically determines the type of captured content and extracts basic metadata without user intervention.

#### Behaviour
- **Text analysis**:
  - Matches URL pattern (`http://`, `https://`) → `url`
  - Plain text → `text` (further classified as `plain`, `markdown`, or `code` by heuristics)
- **File analysis**:
  - Magic number inspection first, extension fallback
  - Image formats (png, jpg, gif, webp) → `image`
  - PDF, zip, txt, md, binary → `file` with subtype
- Type detection runs synchronously at capture time; enrichment runs asynchronously in the background pipeline

#### Acceptance Criteria
- Pasting `https://example.com` → stored as `url` type, not `text`
- Dropping `document.pdf` → stored as `file` with `kind: 'pdf'`
- Dropping `photo.png` → stored as `image`, not `file`
- Type detection is deterministic: same input always yields same type

### F-03 — Chronological Feed (Universal)

**Priority:** P0
**Applies to:** Web UI, CLI (`inbox list`)

#### Description
All captured items are displayed in a strictly chronological, append-only feed: oldest at the top, newest at the bottom.

#### Behaviour
- Feed loads all items on launch, sorted by `capturedAt` ascending
- New captures append at the bottom and auto-scroll into view
- Scroll position is preserved between sessions
- Each block shows:
  - Type icon (16px, muted color)
  - Timestamp (smart formatting: "14:32", "Mar 12 · 09:01", "2025 Mar 3 · 09:01")
  - Content (type-specific rendering)
  - Optional metadata/enrichment (loaded lazily)

#### Block Rendering

| Type | Header | Content | Footer |
|------|--------|---------|--------|
| `text` | `Article` icon | Body text, truncated at 5 lines | "Show more/less" if truncated |
| `url` | `Link` icon + hostname | Full URL in accent, clickable | Title/description if enriched |
| `image` | None (image is self-evident) | Inline image, constrained to feed width | Dimensions if extracted |
| `file` | Type-specific icon + filename | "Size: " + formatted size | None |

#### Acceptance Criteria
- Feed renders 10,000 items without crashing (virtualized if needed)
- Auto-scroll to newest on capture
- Timestamps follow smart formatting table
- Block hover: subtle border highlight, no animation
- Empty state: centered hint text "Paste, drop, or type to capture something"

### F-04 — Persistent Storage (Server)

**Priority:** P0
**Applies to:** Server, Database

#### Description
All captured items are stored in a durable database. Raw content is kept immutable and separate from computed metadata and enrichment.

#### Data Contract
The storage system must enforce a strict separation between:

- **Raw data**: the sacred, unmodified input captured from the user. This is immutable and must never be altered, truncated, or enriched in place.
- **Capture metadata**: type-specific facts computed synchronously during capture (e.g., detected subtype, file size, MIME type). Derived from raw data, but stored separately.
- **Background enrichment**: data produced asynchronously by background jobs (e.g., fetched title, summary, tags, embeddings). Optional and recoverable if lost or recomputed.

This separation guarantees that raw content is always recoverable in its original form, regardless of processing failures, schema changes, or enrichment pipeline outages.

#### Behaviour
- Text/URLs stored inline in PostgreSQL
- Images/files stored as blob references (filesystem or object storage)
- Schema migrations are versioned and reversible
- Raw data is write-once, read-only from the application's perspective

#### Acceptance Criteria
- Capture and immediate kill -9 the server → item is present on restart (WAL committed)
- Raw field matches original input exactly
- Database supports 1M+ items with sub-second feed queries
- Raw data is accessible even if enrichment pipeline fails or is disabled

### F-05 — Semantic Search (Server + All Frontends)

**Priority:** P0
**Applies to:** Server, CLI, Web UI

#### Description
Users can search their captured items using natural language. The system finds semantically similar content even when keywords don't match exactly.

#### Data Model (Search-Specific Fields)
```
Item (search extension)
  embedding: vector(384) — nullable, generated async by background pipeline
  searchVector: tsvector — for PostgreSQL full-text search, auto-updated via trigger
```

#### Behaviour
- **CLI**: `inbox search "notes about NixOS configuration"`
- **Web**: Search bar at top of feed; live results as user types
- **API**: `GET /items/search?q=...&limit=20`
- Search operates on:
  - Full-text index (exact word matching, fast)
  - Vector index (semantic similarity, slower but fuzzier)
  - Combined ranking: FTS for recall, vector for re-ranking
- Results show match context (snippet around matched terms)

#### Acceptance Criteria
- Searching "NixOS setup" finds items containing "Nix configuration guide"
- Search returns ranked results with relevance score
- Empty query returns all items (feed view)
- Results update in <300ms for <100k items
- Full-text search returns results in <100ms
- Vector similarity search returns top-k in <200ms
- CLI search works offline if local cache is present (future enhancement)

### F-06 — Automatic Categorization & Tagging (Server)

**Priority:** P1
**Applies to:** Server

#### Description
The system automatically assigns categories and tags to items based on content analysis, without user input.

#### Behaviour
- Background job analyzes `text` and supported `file` items (PDF, md, txt)
- Generates categories: broad buckets (e.g., "Dev", "Design", "Reference", "Personal")
- Generates tags: specific topics (e.g., "NixOS", "React", "Typography")
- User can override auto-tags; manual tags take precedence
- Categories/tags feed into search and browse filters

#### Acceptance Criteria
- A URL about CSS Grid gets tagged `[css, layout, web-dev]`
- A PDF about machine learning gets categorized "ML/AI"
- User can add manual tags via CLI or Web UI
- Auto-tags are suggestions, not locked; user can remove them
- Processing happens within 30 seconds of capture for text items

---

### F-07 — Content Summarization (Server)

**Priority:** P1
**Applies to:** Server, Web UI

#### Description
Long text items, PDFs, and web pages get automatic one-paragraph summaries.

#### Behaviour
- Triggered for text >500 words, PDFs, and URLs with fetchable content
- Background job generates summary via local LLM (no external API calls by default)
- Summary stored in `enrichment.summary`
- Displayed below item content in feed, collapsed by default
- User can opt out globally or per-item

#### Acceptance Criteria
- A 2000-word article gets a 2-sentence summary
- Summary appears in search index (searching summaries finds items)
- User toggle "Auto-summarize" in settings; off by default
- Failed summarization is silent (no error UI); item remains fully usable

#### Behaviour

#### Acceptance Criteria
- `inbox add` returns item ID in <200ms
- `--json` flag produces valid JSON

- Works on Linux, macOS, and Windows (via WSL)

### F-15 — API — Items

**Priority:** P0
**Applies to:** Server

#### Description
RESTful endpoints for creating, listing, and retrieving items.

#### Endpoints
```
POST   /v1/items              # Create item (multipart for files)
GET    /v1/items              # List items (paginated, filterable)
GET    /v1/items/:id          # Get single item
```

#### Authentication
- **v0**: Single-user mode, no auth (localhost/self-hosted only)
- **v1**: API key header (`X-API-Key`)
- **v2**: JWT-based auth (OAuth2, passkey, or similar)

#### Acceptance Criteria
- All endpoints return JSON, accept JSON
- Pagination with `cursor` or `offset/limit`
- Consistent error format: `{ error: string, code: string, details? }`
- Rate limiting on `POST /v1/items` (prevent abuse)
- `POST /v1/items` returns `201` with full item representation in <100ms

### F-18 — Background Processing Pipeline (Infrastructure)

**Priority:** P0
**Applies to:** Server

#### Description
An internal job queue system that processes enrichment, embedding generation, categorization, and summarization asynchronously. This is infrastructure that enables F-02 (type detection enrichment), F-05 (search indexing), F-06 (categorization), and F-07 (summarization) — not a user-facing feature on its own.

#### Pipeline Stages
```
Item Created
  ├─→ Stage 1: Type Detection (immediate, <10ms)
  ├─→ Stage 2: Metadata Extraction (async, <1s)
  │     ├─→ URL: fetch title/description
  │     ├─→ Image: extract dimensions
  │     └─→ File: detect subtype, compute hash
  ├─→ Stage 3: Embedding Generation (async, <5s)
  │     └─→ Generate 384-dimensional vector for text/content
  ├─→ Stage 4: Categorization (async, <10s)
  │     └─→ Assign auto-tags and categories
  └─→ Stage 5: Summarization (async, <30s, optional)
        └─→ Generate summary for long content
```

#### Behaviour
- Jobs are queued and processed by workers
- Failed jobs retry with exponential backoff (max 3 retries)
- Dead-letter queue for permanently failed jobs
- Job status visible via `GET /v1/items/:id` (`enrichmentStatus` field)
- Pipeline is idempotent: re-running on same item is safe

#### Acceptance Criteria
- Item is searchable by full-text immediately after creation
- Item is searchable by vector within 10 seconds of creation
- Failed enrichment does not block item availability
- Job queue survives server restart (persistent queue)
- Admin endpoint to retry failed jobs

### F-19 — Backup & Export (All Interfaces)

**Priority:** P1
**Applies to:** CLI, Web UI, Server

#### Description
Users can export their entire vault or subsets of it in portable formats.

#### Formats
- **JSON**: Full export with all metadata and enrichment
- **Markdown**: Convert items to a flat markdown file (great for Obsidian/Notion import)
- **CSV**: Spreadsheet-friendly export for analysis

#### Behaviour
- **CLI**: `inbox export --format json > backup.json`
- **Web**: Settings → Export → choose format → download
- **API**: `GET /v1/export?format=...&since=...`
- Export includes all items the user has access to
- Incremental export supported (`since` parameter)

#### Acceptance Criteria
- JSON round-trip: export → import → identical data
- Markdown export preserves chronological order
- Export of 10,000 items completes in <30 seconds
- Streaming response for large exports (no memory explosion)

## 3. Out of Scope for v0

| Feature | Reason | Target Version |
|---------|--------|----------------|
| Multi-user / authentication | Single-user self-hosted focus | v1 |

## 5. Decisions Log

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| 1 | CLI or Web first? | CLI first | Faster to build; dogfood the API; power-user focus |
| 2 | Auth in v0? | No | Self-hosted single-user; API key in v1 |
| 3 | Embedding model size? | 384-dimensional | Balance of quality vs. speed vs. storage |
| 4 | File storage? | Filesystem (configurable to S3) | Simple for self-hosting; S3 for scale |
| 5 | Job queue? | In-process first, Redis later | Minimize infrastructure for v0 |
| 6 | Web framework? | Lightweight (not Next.js) | SPA calling API; no SSR needed |
| 7 | Search ranking? | FTS + vector re-ranking | FTS for speed, vector for relevance |
| 8 | API style? | REST (not GraphQL) | Simple, cacheable, CLI-friendly |
| 9 | Database? | PostgreSQL + pgvector | Industry standard; vector extension mature |
| 10 | Server language? | Go | Fast, static binary, easy to deploy, shared with CLI |
