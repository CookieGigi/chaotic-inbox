# Chaotic Inbox — Feature Specification v0

**Version:** 0.1.0 (2026-05-23)
**Status:** In Progress
**Author:** OpenCode

---

## 1. Project Overview

Chaotic Inbox is a **personal knowledge capture tool** designed for **zero-friction capture**. The user pastes, drops, or types anything and it is saved immediately with no prompts, no loading spinners, and no organizational decisions required at capture time.

This specification defines the full-stack v0 implementation: a Python/FastAPI backend server, a Python/Typer CLI, and a React web UI that together provide capture, storage, search, enrichment, and management of personal data items.

## 5. Tech Decisions

### 5.1 Decision Log

| # | Decision | Alternatives | Rationale | Trade-offs |
|---|----------|-------------|-----------|------------|
| 1 | **Python for backend** | Rust, Go, Node.js | Direct access to ML/AI ecosystem (sentence-transformers, Ollama, BeautifulSoup). Much simpler async I/O with FastAPI. No ONNX/FFI complexity. | GIL limits CPU parallelism (mitigated by async I/O + background workers) |
| 2 | **FastAPI for server** | Flask, Django, Tornado | Native async, auto OpenAPI, deep Pydantic integration, excellent performance | Slightly more opinionated than Flask |
| 3 | **SQLModel for ORM** | SQLAlchemy raw, Django ORM, Prisma | Single model class serves as both DB table and API schema. Eliminates DTO mapping layer. | Relatively new project (backed by SQLAlchemy 2.0) |
| 4 | **asyncpg for PostgreSQL** | psycopg3, sqlite | Fastest async PostgreSQL driver. Required for SQLAlchemy async mode. | PostgreSQL-only (no SQLite in production) |
| 5 | **Typer for CLI** | Click, argparse, Rich CLI | Type hints for args/options, auto-generated help, shell completion. Built on proven Click foundation. | Less flexible than raw Click for very complex UIs |
| 6 | **httpx for HTTP client** | requests, aiohttp | Modern, HTTP/2, sync+async APIs, type-safe | Slightly larger than requests |
| 7 | **sentence-transformers** | ONNX Runtime, OpenAI API | Native Python, runs locally, no API keys, all-MiniLM-L6-v2 is fast and small (~100MB) | Requires PyTorch/Transformers dependencies |
| 8 | **PostgreSQL + pgvector** | SQLite, MongoDB, Qdrant | Mature, reliable, full SQL + vector search in one database. No extra infrastructure. | Requires PostgreSQL installation |
| 9 | **Alembic for migrations** | raw SQL, Django migrations | Auto-generate from SQLModel metadata. Industry standard. | Requires careful migration review |
| 10 | **uv for package management** | pip, poetry, pdm | Extremely fast resolution and installation. Modern lockfile. | Newer tool, some edge cases |
| 11 | **Zustand for web state** | Redux, React Query | Lightweight, no boilerplate, works well with server-cached state | Manual cache invalidation |
| 12 | **Pydantic Settings for config** | python-dotenv, dynaconf | Type-safe environment variables, `.env` file support, validation | Tied to Pydantic ecosystem |
| 13 | **PostgreSQL job_queue** | Redis + Celery, RabbitMQ, arq | No extra infrastructure. Single database handles everything. | Less robust than dedicated queue for high throughput |
| 14 | **ruff for lint/format** | flake8 + black + isort | Single tool replaces 3. Much faster. | Newer, some rules still maturing |

### 5.2 Python Ecosystem Rationale

**Why Python over Rust for the backend?**

The original architecture specified Rust (axum + sqlx) for the backend. While Rust provides excellent performance and memory safety, the project's core differentiating features — semantic search via embeddings, AI summarization, automatic categorization, and URL metadata extraction — all depend on the Python ML/AI ecosystem.

Using Rust would require:
- ONNX Runtime bindings for embedding generation (complex FFI, limited model support)
- Separate Python sidecar for LLM inference (Ollama, transformers)
- Manual HTTP clients for URL enrichment instead of BeautifulSoup
- More code for the same functionality (Rust's explicitness is a cost for rapid iteration)

Python eliminates these complexities:
- `sentence-transformers` provides embeddings in 3 lines of code
- `ollama` Python client talks to local LLM directly
- `BeautifulSoup4` + `httpx` fetch and parse URL metadata natively
- `FastAPI` + `SQLModel` provide type-safe APIs with minimal boilerplate

The trade-off — Python's GIL and slower raw CPU performance — is acceptable for a single-user self-hosted application where the bottleneck is I/O (database, HTTP requests) rather than CPU.

## 6. Non-Functional Requirements

### 6.1 Performance
- **Capture latency:** < 500ms from user action to persisted item
- **Feed rendering:** < 100ms for 50 items
- **Search response:** < 200ms for full-text search
- **Vector search:** < 500ms for top-k similarity query
- **CLI startup:** < 500ms cold start

### 6.2 Scalability
- **Target:** Single user, thousands of items
- **PostgreSQL:** Handles millions of rows easily
- **Embeddings:** 384-dim vectors, ~1.5KB per item
- **Blobs:** Stored on filesystem, not in database

### 6.3 Reliability
- **Data durability:** PostgreSQL ACID guarantees
- **Soft deletes:** Items recoverable until hard delete
- **Job retry:** Failed enrichment jobs retry up to 3 times
- **Backup:** JSON export + filesystem blob backup

### 6.4 Security
- **Single-user:** No authentication in v0 (local network only)
- **Input validation:** Pydantic models validate all API input
- **SQL injection:** SQLAlchemy parameterized queries prevent injection
- **XSS:** Content sanitized before rendering in web UI
- **CORS:** Restricted to known origins in production

### 6.5 Maintainability
- **Code organization:** Clear separation (routers, services, models, CLI)
- **Type safety:** Full type hints across all Python code (mypy/strict)
- **Testing:** pytest for Python, Vitest for web UI
- **Documentation:** Auto-generated OpenAPI docs from FastAPI
- **Linting:** ruff enforces consistent style

## 7. Data Model Summary

### 7.1 Item States
```
[Capture] → [Persisted] → [Enrichment Jobs Queued] → [Enriching] → [Enriched]
                                              ↓
                                          [Failed] → [Retry]
```

### 7.2 Enrichment Tracks
Each item has independent enrichment tracks:
- **URL track:** `pending → running → done | failed | skipped`
- **Image track:** `pending → running → done | failed | skipped`
- **AI track:** `pending → running → done | failed | skipped`

### 7.3 Search Indexing
- **Full-text:** PostgreSQL tsvector automatically updated via trigger
- **Vector:** Embedding generated by background job, stored in pgvector column
- **Hybrid:** Combine FTS relevance score + vector cosine similarity for ranking

### B. Glossary
| Term | Definition |
|------|-----------|
| **Item** | A single captured piece of content (text, URL, image, file) |
| **Block** | A UI component representing an item in the feed |
| **Feed** | The chronological list of all items |
| **Enrichment** | Background processing that adds metadata (title, dimensions, summary, tags) |
| **Track** | An independent enrichment pipeline (URL, image, AI) |
| **Embedding** | A vector representation of content for semantic search |
| **pgvector** | PostgreSQL extension for storing and querying vector embeddings |
| **SQLModel** | Library combining SQLAlchemy ORM with Pydantic validation |
